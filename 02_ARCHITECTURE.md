# 🏗️ สถาปัตยกรรม TimeCareHub

**อัปเดตล่าสุด:** 2026-07-17 — ตรงกับโค้ดจริงแล้ว

## Tech Stack

| ชั้น | ใช้อะไร | ทำไม |
|---|---|---|
| Frontend | **HTML + JS ธรรมดา** (ไม่มี framework) | MVP เอาไปโชว์ — ไม่ต้อง build โยนไฟล์ขึ้น server ได้เลย |
| แผนที่ | **Leaflet + OpenStreetMap** | ฟรี ไม่ต้องผูกบัตรเครดิต ไม่ต้องมี API key |
| Backend | **Node.js 22 + Express** | stack ที่พี่ดิวถนัด |
| Database | **MySQL 8.4** | มีอยู่บนเครื่องแล้ว |
| Auth | **JWT ใน httpOnly cookie + bcrypt** | JS ในหน้าเว็บอ่าน token ไม่ได้ → กัน XSS ขโมย token |
| แชท | **HTTP polling ทุก 3 วินาที** | MVP ไม่ต้องใช้ WebSocket — ง่ายกว่า พอเพียงสำหรับเดโม |
| แจ้งเตือน | **polling ทุก 15 วินาที** → ตัวเลขแดงบนแท็บ | ไม่ใช่ push จริง แต่พอให้รู้ว่ามีของใหม่ |
| Process manager | **pm2** (`timecarehub-8091`) | รีบูตเครื่องแล้วขึ้นเอง |

> **ไม่ได้ใช้:** React, WebSocket, Google Maps, payment gateway, Firebase — ตัดออกหมดเพื่อให้ MVP เสร็จเร็ว

## ⭐ ระบบจ้างงาน — มี **2 แบบ** วิ่งคู่กัน

นี่คือสิ่งที่เอกสารรุ่นแรกไม่มี เพิ่มเข้ามาทีหลัง (`db/002_direct_hire.sql`)

| | **แบบโพสงาน** (`hire_type='open'`) | **แบบจ้างตรง** (`hire_type='direct'`) |
|---|---|---|
| ใครเริ่ม | ผู้ว่าจ้างโพสงานทิ้งไว้ | ผู้ว่าจ้างเดินดูโปรไฟล์แล้วยิงตรงไปหาคนที่ถูกใจ |
| ปักหมุด GPS | ✅ ต้องปัก | ❌ ไม่ต้อง (`lat`/`lng` เป็น NULL ได้) |
| ใครกดก่อน | แคร์กิฟเวอร์กดขอรับ (หลายคนได้) | ผู้ว่าจ้างส่งคำขอไปหา 1 คน |
| ใครเลือก | **ผู้ว่าจ้าง**เลือกจากคนที่กดขอ | **แคร์กิฟเวอร์**กดรับหรือปฏิเสธ |
| วงจรสถานะ | `open → matched → done` | `offered → matched → done`<br>`offered → declined` |
| route | `/api/jobs` | `/api/hires` + `/api/caregivers` |

> **เก็บในตาราง `jobs` ตารางเดียวกัน** แยกด้วยคอลัมน์ `hire_type`
> ทำแบบนี้เพื่อให้ **แชท / ปิดงาน / รีวิว ใช้โค้ดชุดเดียวกันได้ทั้ง 2 แบบ** ไม่ต้องเขียนซ้ำ

### หน้าเว็บทำงานยังไง

ทุกหน้าโหลด `frame.js` ก่อนเสมอ → `buildFrame()` จะ:
1. เรียก `/api/auth/me` — ไม่ผ่าน = เด้งกลับหน้า login (auth guard)
2. สลับ `active_role` ให้ตรงกับหน้าที่เปิด (กัน state ค้าง)
3. สร้าง header + แถบเมนูล่าง แล้วเรียกฟังก์ชัน render ของแท็บแรก
4. ตั้ง polling เช็คแจ้งเตือนทุก 15 วิ

ตัวแปรร่วมที่ทุกหน้าใช้: `ME` (ผู้ใช้ปัจจุบัน), `view` (`<main>`), `api()`, `toast()`, `esc()`, `ICONS`

## ⭐ ตรรกะสำคัญที่สุด: การเปิดเผยพิกัด 2 ระดับ

**ไฟล์:** `src/geo.js` — **นี่คือหัวใจของแอพ ห้ามถอด**

| ใครดู | เห็นอะไร |
|---|---|
| เจ้าของงาน (ผู้ว่าจ้าง) | 📍 ทุกอย่าง |
| แคร์กิฟเวอร์ที่ **ถูกเลือกแล้ว** | 📍 ทุกอย่าง |
| แคร์กิฟเวอร์ที่ **ยืนยันตัวตนแล้ว** (`kyc_status='approved'`) | 📍 พิกัดเป๊ะ + ที่อยู่เต็ม + เบอร์โทร |
| แอดมิน | 📍 ทุกอย่าง |
| **แคร์กิฟเวอร์ที่ยังไม่ยืนยันตัวตน** | ⭕ **วงกลมรัศมี 800 ม. เท่านั้น** — ไม่มีที่อยู่ ไม่มีเบอร์ |

### ทำไมต้องมี

ที่อยู่บ้านที่มีผู้สูงอายุอยู่ลำพัง = ข้อมูลอ่อนไหวสูงมาก
ถ้าเปิดให้ทุกคนที่สมัครเข้ามาเห็น เท่ากับประกาศให้คนไม่หวังดีรู้ว่าบ้านไหนมีคนแก่อยู่คนเดียว

ผลพลอยได้: เป็นแรงจูงใจให้แคร์กิฟเวอร์รีบยืนยันตัวตน

### วิธีเบลอ — ปัดลงกริด ไม่ใช่สุ่ม

```js
const GRID = 0.009;   // ~1 กม.
lat = Math.round(lat / GRID) * GRID;
lng = Math.round(lng / GRID) * GRID;
```

**ทำไมไม่สุ่ม:** ถ้าสุ่มจุดใหม่ทุกครั้งที่เรียก API คนไม่หวังดีจะกด refresh 100 ครั้ง
เก็บพิกัดมาเฉลี่ย แล้วได้ตำแหน่งจริงกลับมา

วิธีปัดลงกริด → **พิกัดเดิมได้จุดเบลอเดิมเสมอ** ไม่ว่าจะ refresh กี่ครั้ง
สิ่งที่รู้ได้มากที่สุดคือ "อยู่ในช่องกริด 1 กม. นี้" — เดาตำแหน่งจริงไม่ได้

## ⚠️ KYC ตอนนี้เป็น "โหมดเดโม"

**ไฟล์:** `src/routes/kyc.js`

| | ที่ออกแบบไว้ | **ที่เป็นอยู่จริงตอนนี้** |
|---|---|---|
| ขั้นตอน | อัปบัตร ปชช. + เซลฟี่ → เข้าคิว → แอดมินกดอนุมัติ | **กดปุ่ม "ยืนยันตัวตน" → approved ทันที** |
| ไฟล์อัปโหลด | multer เก็บลงดิสก์ | ไม่มี |
| หน้าแอดมิน | คิวรออนุมัติ + ดูรูปบัตร | รายชื่อแคร์กิฟเวอร์ + ปุ่มเพิกถอน |

**ทำไม:** เดโมยาก — ต้องหารูปมาอัป แล้วต้องสลับไปล็อกอินเป็นแอดมินอีกรอบ

**⭐ สิ่งที่ไม่ได้ถอดออก:** `kyc_status` ยังคุม GPS 2 ระดับเหมือนเดิมทุกประการ
คอลัมน์ `kyc_id_card` / `kyc_selfie` ยังอยู่ครบใน DB → **เปิดของจริงกลับมาได้ทันที**

## ⏸ ระบบให้ดาว/รีวิว — ปิดอยู่

`server.js` คอมเมนต์บรรทัด `app.use('/api/reviews', ...)` ไว้ (ตกลงกัน 2026-07-14 — เดโมยังไม่ต้องใช้)

- ตาราง `reviews` + `src/routes/reviews.js` **ยังอยู่ครบ** — เอาคอมเมนต์ออกก็กลับมาทันที
- `caregiver_profiles.rating_avg` / `rating_count` ยังมี แต่เป็น 0 หมด เพราะไม่มีใครให้ดาวได้
- หน้าเว็บที่โชว์ ⭐ จะขึ้น 0.0 (0) เสมอ

## Middleware เช็คสิทธิ์ (`src/auth.js`)

| ตัว | ทำอะไร |
|---|---|
| `requireAuth` | ต้องล็อกอิน — **โหลด user สดจาก DB ทุกครั้ง** ไม่เชื่อข้อมูลใน token<br>(เพราะสถานะ KYC เปลี่ยนได้ตลอด ถ้าเชื่อ token เก่า อนุมัติแล้วจะไม่มีผลจนกว่าจะล็อกอินใหม่) |
| `requireAdmin` | เฉพาะแอดมิน — ใช้กับหน้าแอดมิน |
| `requireApprovedCaregiver` | เฉพาะแคร์กิฟเวอร์ที่ `kyc_status='approved'` — ใช้กับการกดขอรับงาน |

## จุดที่ระวังเรื่องความปลอดภัยไว้แล้ว

| เรื่อง | ทำอะไรไว้ |
|---|---|
| SQL Injection | ใช้ prepared statement (`?`) ทุกที่ ไม่มีการต่อ string |
| XSS | `esc()` ใน `frame.js` escape ทุกข้อมูลจากผู้ใช้ก่อนใส่ HTML |
| ขโมย token | JWT อยู่ใน httpOnly cookie — JS อ่านไม่ได้ |
| รหัสผ่าน | bcrypt (cost 10) — ไม่เคยเก็บ plain text |
| บอกใบ้ว่าอีเมลไหนมีในระบบ | login ผิด → ข้อความเดียวกันทั้งกรณีไม่มีอีเมล และรหัสผิด |
| **เลขบัตรประชาชนรั่ว** | **`/api/profile` เป็น route เดียวที่ส่งออกได้ และส่งให้เจ้าของบัญชีเท่านั้น**<br>route อื่น **ห้าม `SELECT u.*`** ต้องระบุคอลัมน์เอง |
| **ยกตัวเองเป็นแอดมิน** | `/api/profile` ใช้ **allowlist** ของคอลัมน์ที่แก้ได้ — `is_admin`, `email`, `password_hash` ไม่อยู่ในลิสต์ |
| รูปบัตรประชาชนรั่ว | เก็บนอก `public/` เข้า URL ตรงไม่ได้ |
| กดขอรับงานซ้ำสแปม | `UNIQUE KEY (job_id, caregiver_id)` ใน DB |
| ส่งคำขอจ้างซ้ำ | เช็ค `status='offered'` ของคู่เดิมก่อน INSERT |
| แย่งกันเลือกคนพร้อมกัน | ใช้ transaction + `SELECT ... FOR UPDATE` ตอนเลือกแคร์กิฟเวอร์ |
| เลขบัตรมั่ว | ตรวจหลักตรวจสอบ (checksum) ของบัตรไทยทั้งฝั่งหน้าเว็บและฝั่ง server |

### ⚠️ ที่ยังไม่ได้ทำ (รับได้สำหรับ MVP เดโม แต่ต้องทำก่อนขึ้นจริง)

- ไม่มี rate limit — ยิง login เดาสุ่มรัว ๆ ได้
- ไม่มี CSRF token (cookie เป็น `sameSite: lax` ช่วยได้ระดับหนึ่ง)
- SSL เป็น self-signed
- port 8091 เปิดวง LAN ตรง ๆ (ตั้ง `HOST=127.0.0.1` ใน `.env` เพื่อปิด)
- **KYC เป็นโหมดเดโม** — ใครกดปุ่มก็ผ่าน
- **repo เป็น public** — เอกสารในนี้มี IP + ผังเครื่อง
