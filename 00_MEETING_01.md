## 1. สิ่งที่เข้าใจจากที่พี่ดิวเล่า

TimeCareHub = แอพจับคู่ **ผู้ว่าจ้าง** กับ **แคร์กิฟเวอร์ (ผู้ดูแลผู้สูงอายุ)**

**Flow ที่เล่ามา:**

1. เปิดแอพ → login
2. หลัง login ให้ **เลือกบทบาท**
   - "ฉันต้องการจ้างแคร์กิฟเวอร์" (ผู้ว่าจ้าง)
   - "ฉันเป็นแคร์กิฟเวอร์" (ผู้รับงาน)
3. **ฝั่งผู้ว่าจ้าง:** โพสงานดูแลผู้สูงอายุ + ปักหมุด GPS ตำแหน่งงาน
4. **ฝั่งแคร์กิฟเวอร์:** เห็นงานในพื้นที่ใกล้ตัว → กด "ขอรับงาน"
5. ทั้งสองฝ่าย **แชทคุยกัน** → ตกลงระยะเวลางาน + ราคา
6. (ยังไม่ได้คุย: หลังตกลงแล้วเกิดอะไรต่อ — จ่ายเงิน? รีวิว? ปิดงาน?)

## 4. ประเด็นที่ AI ท้วง (รอพี่ดิวเคาะ)

### ⚠️ GPS เป๊ะ vs ความปลอดภัยผู้สูงอายุ

พี่ดิวเลือก "เห็นพิกัดเป๊ะ" ซึ่งพอมี KYC + แอดมินอนุมัติ ความเสี่ยงลดลงเยอะแล้ว
เพราะคนเห็นงานคือคนที่ผ่านการตรวจบัตรประชาชนแล้วเท่านั้น

**AI เสนอให้ผูกกฎนี้เข้าไปในระบบเลย:**

| สถานะแคร์กิฟเวอร์ | เห็นอะไร |
|---|---|
| ผ่าน KYC + แอดมินอนุมัติแล้ว | 📍 **พิกัดเป๊ะ + ที่อยู่เต็ม** (ตามที่พี่ดิวต้องการ) |
| สมัครแล้ว รออนุมัติ / ยังไม่ทำ KYC | ⭕ **วงกลมคร่าว ๆ** (เห็นแค่ว่า "แถวนี้มีงาน 5 งาน") |

**เหตุผล:** ที่อยู่บ้านที่มีคนแก่อยู่ลำพัง = ข้อมูลอ่อนไหวมาก ถ้าใครโหลดเว็บก็เห็นหมด
เท่ากับประกาศให้โจรรู้ + ยังเป็นแรงจูงใจให้แคร์กิฟเวอร์รีบทำ KYC ด้วย

> สถานะ: **รอพี่ดิวตอบว่าโอเคไหม**

## 6. สรุปสเปค MVP (ตกลงแล้ว)

### Stack
```
Frontend : เว็บ (Vanilla/React — รอเคาะอีกที)
Map      : Leaflet + OpenStreetMap
Backend  : Node.js + Express
Database : MySQL
Deploy   : serverlive 192.168.1.35 (nginx + pm2)
Mobile   : ❌ ยังไม่ทำ — Flutter ไว้ทีหลัง
```

### Flow หลัก
```
1. สมัคร/Login (อีเมล + รหัสผ่าน)
2. เลือกโหมด: [ต้องการจ้าง] หรือ [เป็นแคร์กิฟเวอร์]  ← สลับได้ตลอด บัญชีเดียว

── ฝั่งผู้ว่าจ้าง ──
3. โพสงาน: อาการผู้สูงอายุ + สิ่งที่ต้องทำ + ระยะเวลา + งบ
4. ปักหมุด GPS ตำแหน่งบ้าน
5. รอแคร์กิฟเวอร์กดขอรับ → เห็นรายชื่อ + โปรไฟล์ + ดาว
6. เลือก 1 คน → แชทตกลงเวลา/ราคา
7. งานเสร็จ → จ่ายเงินกันเอง (นอกระบบ) → ให้ดาว/รีวิว

── ฝั่งแคร์กิฟเวอร์ ──
3. ทำ KYC (อัพบัตรประชาชน + เซลฟี่) → รอแอดมินอนุมัติ
4. อนุมัติแล้ว → เห็นงานบนแผนที่ + ค้นหาตามระยะ
5. กดขอรับงาน → รอผู้ว่าจ้างเลือก
6. ถูกเลือก → แชท → ทำงาน → รับเงินสด → ถูกรีวิว

── ฝั่งแอดมิน (เว็บ) ──
- ดูคิว KYC ที่รออนุมัติ → กดอนุมัติ / ปฏิเสธ
```

## 8. โมเดลรายได้ — AI ท้วงไว้ (ไม่ด่วน แต่ควรคิด)

MVP ไม่เก็บเงิน = โอเค เอาไปโชว์ก่อนถูกแล้ว
แต่พออนาคตจะเก็บเงิน จะเจอปัญหาคลาสสิกของแอพจับคู่ทุกตัวคือ **"คู่ค้าหนีออกนอกแอพ"**

เพราะเราให้เขาแลกเบอร์/คุยกันในแชท แล้วจ่ายเงินกันเอง →
ครั้งแรกใช้แอพ ครั้งต่อไปโทรหากันตรง ๆ แอพไม่ได้อะไรเลย

**ทางแก้ที่แอพอื่นเขาใช้:**
- ระบบตั๋ว (เก็บตอนแคร์กิฟเวอร์กดขอรับงาน — เก็บก่อนที่เขาจะหนีได้)
- Escrow (เก็บ % จากเงินที่วิ่งผ่านระบบ — แต่ต้องมี payment gateway)
- ทำให้การอยู่ในแอพคุ้มกว่า: ประกัน, ระบบเช็คอิน-เช็คเอาท์, ปุ่มฉุกเฉิน, ประวัติการทำงาน

> ไม่ต้องตัดสินใจตอนนี้ แค่ **ออกแบบ DB เผื่อไว้** จะได้ไม่ต้องรื้อทีหลัง

| 4 | logo แก้ใหม่ + TimeCareHub → TIMECAREHUB | **พี่ดิวเลือก "วาด logo ใหม่เป็น SVG"** แต่พอเปิดไฟล์จริงพบว่า `logo-mark.png` (หัวใจ + คน 2 คน + นาฬิกาทราย) **เป็นสีฟ้าตรงกับ #4FC4FF อยู่แล้ว และสวยกว่าที่จะวาดใหม่** → เก็บตัวสัญลักษณ์ไว้ใช้ต่อ เอาขึ้นหัวเว็บด้วย (พื้นขาวกลม ไม่งั้นจมกับหัวเว็บสีฟ้า) แล้วทำ **wordmark เป็นตัวหนังสือไล่เฉด** แทน — คมทุกขนาดจอเหมือนกัน ไม่ต้องมีไฟล์เพิ่ม<br>ข้อความทั้งเว็บเป็น `TIMECAREHUB` แล้ว |
| 5 | "ที่อยู่เต็ม" → "ค้นหา" + ปักหมุดอยู่บล็อกเดียวกัน | รวมเป็นการ์ดเดียว label = "ค้นหา" · ยังบันทึกเป็น `address` เหมือนเดิม (แยกคนละกล่องแล้วคนกรอกที่อยู่เสร็จก็เลื่อนผ่านแผนที่ไป นึกว่าจบแล้ว) |

| 7 | "งานของฉัน" → "ภารกิจของฉัน" | ✅ ทั้ง 2 ฝั่ง |
